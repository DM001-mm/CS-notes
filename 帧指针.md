# 必要性

为什么要用 FP 的偏移量，而不是直接使用整个栈的绝对地址？

因为函数代码是复用的，但函数每次被调用时，都会创建一个新的函数实例，也就是新的栈帧。这个栈帧的实际内存地址只有运行时才知道，编译器在编译阶段无法提前写死某个局部变量的绝对地址。

因此，编译器会把函数栈帧中的参数、局部变量、保存寄存器等内容，安排成相对于某个稳定基准点的固定偏移量。这个稳定基准点通常就是 FP。

例如：

```asm
[fp + 8]   ; 某个参数
[fp - 4]   ; 某个局部变量
```

这样一来，只要运行时确定了当前函数实例的 FP，就可以通过固定偏移量访问这个函数栈帧中的局部状态。

---

# 重要性

在函数调用时，新函数会建立自己的栈帧，并通常把调用者的 FP 保存到新栈帧里。这样，当前真正活跃的 FP 寄存器只有一个，但栈中保存着很多旧 FP 的值。

这些旧 FP 形成了一条链：

```text
当前函数栈帧 -> 调用者栈帧 -> 调用者的调用者栈帧 -> main 栈帧
```

这就像树中的子节点保存父节点指针，可以沿着父节点一路向上回溯。

因此，调试器可以从当前 FP 开始，读取旧 FP 和返回地址，一层一层还原函数调用链，这就是栈追踪。

栈追踪对调试和漏洞修复很重要，因为它不仅能告诉我们程序崩在哪里，还能告诉我们程序是沿着哪条调用路径走到这里的。它可以帮助定位错误传播路径、分析崩溃原因、判断是否存在栈破坏、返回地址异常等问题。

---

# 缺陷

FP 不是程序正确运行的唯一方式。现代编译器可以省略 FP，改用 SP 相对寻址或者 unwind 信息来管理栈帧。

保留 FP 的好处是调试、栈追踪、性能分析更方便；缺点是它会占用一个寄存器，并带来少量函数进入和返回时的额外指令开销。

所以一些追求性能的编译配置会选择省略帧指针，但很多需要稳定调试、线上崩溃分析、性能采样的项目也会选择保留帧指针。
