# 写时拷贝 Copy-On-Write，COW

## 为什么需要写时拷贝

现在考虑两个场景。

第一个场景是：

> 两个应用程序拥有很多相同的内存数据，比如加载了相同的动态链接库。

如果这些相同的数据页在物理内存中只保存一份，然后把这个物理页以只读或者写时拷贝的方式映射给多个应用程序，那么就能够显著节约物理内存资源。

第二个场景是：

> 在 Linux 操作系统中，一个应用程序可以通过 `fork()` 系统调用创建子进程。

从程序语义上看，fork 刚完成时，父进程和子进程应该拥有完全一样的内存数据和地址空间。

但是如果 fork 时直接复制父进程的全部物理页，代价会很大。尤其是很多程序 fork 之后马上 exec，原来复制的内存很快就会被丢弃。

所以操作系统使用写时拷贝技术。

写时拷贝综合了上面两个场景的思想：

```text
多个进程拥有相同内容的页面时，先共享同一个物理页；
只有当某个进程真的要修改这个页面时，才复制出新的物理页。
```

这样既能节约物理内存，又能减少 fork 时的时间开销。

---

# 核心思想

写时拷贝的核心思想是：

> 读的时候共享，写的时候才拷贝。

也就是说：

```text
能不复制就先不复制。
多个进程先共享同一个物理页。
等某个进程真的要写这个页面时，操作系统再给它复制一份。
```

从程序视角看，父子进程好像一开始就拥有各自独立的内存。

但从物理内存视角看，父子进程一开始其实共享同一批物理页。

只有真正发生写入时，物理页才会被复制。

---

# CPU 硬件实现基础

写时拷贝依赖虚拟内存机制，尤其依赖页表项中的权限位。

页表项中不只有物理页地址，还包括一些属性位，例如：

```text
present 位：页面是否存在
read / write 位：是否允许读写
execute 位：是否允许执行
user 位：用户态是否可以访问
```

写时拷贝主要用到的是：

```text
读 / 写权限位
```

操作系统会把多个进程共享的 COW 页面暂时设置成只读。

这样一旦某个进程尝试写入这个页面，MMU 在查询页表时就会发现：

```text
这个页面存在，但是没有写权限。
```

于是 CPU 会触发 page fault 异常，陷入内核，让操作系统的异常处理函数接管。

# 逻辑过程

以 fork 为例。

父进程原来有一些虚拟页：

```text
父进程虚拟页 A -> 物理页 P1
父进程虚拟页 B -> 物理页 P2
父进程虚拟页 C -> 物理页 P3
```

调用 `fork()` 后，操作系统不会立刻复制 P1、P2、P3。

而是让子进程的页表也指向这些相同的物理页：

```text
父进程虚拟页 A -> 物理页 P1
子进程虚拟页 A -> 物理页 P1

父进程虚拟页 B -> 物理页 P2
子进程虚拟页 B -> 物理页 P2
```

同时，操作系统会把这些原本可写的页面暂时设置成只读，并在内核数据结构中记录：

```text
这些页面是 COW 页面。
```

这时父子进程的页表指向同一批物理页。

从物理内存角度看，它们共享内存页。

但从程序语义角度看，它们仍然应该拥有彼此独立的地址空间。

---

# 写入时发生什么

当父进程或子进程尝试写入某个 COW 页时，过程如下：

```text
1. 进程执行一条普通写指令。
2. MMU 查询页表，发现该页面存在，但是页表项不可写。
3. CPU 触发 page fault 异常，陷入内核。
4. 操作系统的 page fault 异常处理函数开始工作。
5. 内核检查该虚拟地址对应的内存区域。
6. 内核发现这不是非法访问，而是写入 COW 页面。
7. 内核分配一个新的物理页。
8. 把旧物理页中的内容复制到新物理页。
9. 修改当前进程的页表项，让这个虚拟页重新映射到新的物理页。
10. 把当前进程对应页表项的权限设置成可读可写。
11. 使旧的 TLB 缓存项失效。
12. 回到异常发生处，重新执行刚才那条写指令。
```

例如，子进程写入虚拟页 A 之后，映射关系可能变成：

```text
父进程虚拟页 A -> 原物理页 P1
子进程虚拟页 A -> 新物理页 P4
```

这样父进程和子进程之后再修改各自的页面，就不会互相影响。

---

# 注意：COW 不是“到写入时才映射物理地址”

COW 页在 fork 之后已经有物理地址映射了。

例如：

```text
父进程虚拟页 A -> 物理页 P1，只读，COW
子进程虚拟页 A -> 物理页 P1，只读，COW
```

写入时不是“才映射物理地址”，而是：

```text
分配新的物理页 P4
把 P1 的内容复制到 P4
修改当前进程的页表项
让当前进程虚拟页 A 重新映射到 P4
并设置为可读可写
```

所以更准确地说：

> COW 写入时，操作系统不是第一次建立映射，而是重新建立当前进程虚拟页到新物理页的映射。

---

# 注意：COW 的“只读”是假只读吗？

可以说是假只读，但要分清楚它假在哪里。

从硬件页表权限看，它是真的只读：

```text
页表项 writable = 0
```

所以 MMU 真的会拦截写操作。

但是从操作系统语义看，它是假只读：

```text
这个页面本来应该允许进程写。
只是操作系统为了实现延迟复制，暂时把它设置成只读。
```

从进程最终看到的效果看，它仍然等价于可写：

```text
进程第一次写入
    ↓
触发 page fault
    ↓
内核复制物理页
    ↓
当前进程获得自己的可写副本
    ↓
写入成功
```

所以可以这样记：

```text
硬件权限：真只读
操作系统意图：假只读
进程最终语义：可写
```

COW 的只读不是目的，而是手段。

它是操作系统故意设置的一个“陷阱”，目的是在第一次写入时让 CPU 陷入内核，从而让操作系统有机会完成复制。

---

# 为什么这里没有矛盾

一开始容易产生矛盾，是因为把两个概念混在了一起：

```text
页表项只读
页面语义上不能写
```

这两个概念不是一回事。

对于真正的只读页，例如动态链接库的代码段：

```text
页表项只读
页面语义上也不能写
写了就是非法访问
```

而对于 COW 页：

```text
页表项暂时只读
页面语义上仍然应该可写
写入时触发 page fault
内核复制物理页后恢复可写
```

所以：

> 不是所有“写只读页”都会触发写时拷贝。  
> 只有内核判断这个页面是 COW 页时，写只读页才会触发复制。

MMU 只负责发现：

```text
进程正在写一个页表项不可写的页面。
```

但是 MMU 不知道这个页面为什么只读。

真正判断的是操作系统：

```text
如果是真只读页，说明非法访问。
如果是 COW 页，说明需要复制物理页。
```

# 动态链接库中的两种页面

动态链接库中也要分情况看。

## 只读部分

比如动态库的代码段 `.text`。

这部分本来就不应该被修改，所以可以直接被多个进程共享。

```text
进程 A 的虚拟页 -> 动态库代码物理页 P，只读
进程 B 的虚拟页 -> 动态库代码物理页 P，只读
```

如果进程强行写入，就是非法访问。

这类页面是：

```text
真只读共享
```

## 可写部分

动态库也可能有可写数据部分。

这些页面初始内容可能相同，但是不同进程之后可能会分别修改它们。

所以可以先用 COW 方式共享。

```text
进程 A 的虚拟页 -> 物理页 P，只读，COW
进程 B 的虚拟页 -> 物理页 P，只读，COW
```

如果进程 A 修改该页面，操作系统就复制出一份新的物理页给进程 A。

```text
进程 A 的虚拟页 -> 新物理页 P2，可读可写
进程 B 的虚拟页 -> 原物理页 P，只读，COW
```

所以动态库可以这样理解：

```text
只读部分：直接只读共享
可写部分：初始共享，写时拷贝
```

# COW 中的 page fault 不一定是传统意义上的缺页

page fault 不一定表示页面不在物理内存中。

page fault 可能有多种情况：

```text
页面不在内存中，需要从磁盘调入
页面存在，但是访问权限不够
写入 COW 页，需要复制物理页
非法访问，需要终止进程
```

在 COW 中，页面通常已经在物理内存中。

触发 page fault 的原因不是页面不存在，而是：

```text
页面存在，但是当前页表项没有写权限。
```

所以 COW 中的 page fault 更像是一种权限异常，而不是传统意义上的磁盘换页。

---

# COW 不是系统调用

写 COW 页时，进程并不是主动调用系统调用。

系统调用是程序主动请求进入内核，例如：

```c
read();
write();
fork();
exec();
```

而 COW 的触发过程是：

```text
用户程序执行普通写指令
    ↓
MMU 发现写权限不足
    ↓
CPU 自动触发 page fault 异常
    ↓
CPU 陷入内核
    ↓
执行 page fault 异常处理函数
```

所以 COW 属于异常处理，不是系统调用。

可以说：

> 写 COW 页时，CPU 因为 page fault 异常陷入内核。

不要说：

> 写 COW 页时发生了系统调用。

---

# TLB 的处理

操作系统完成 COW 后，主要修改的是页表项。

但是 TLB 中可能还缓存着旧的地址映射，例如：

```text
虚拟页 A -> 原物理页 P1，只读
```

因此，内核修改页表后，需要使对应的旧 TLB 项失效。

之后 CPU / MMU 再访问这个虚拟地址时，会根据新的页表项重新建立映射。

所以更准确地说：

> 操作系统修改页表，并刷新或失效旧的 TLB 项；新的 TLB 项通常会在后续访问时由硬件重新填充。

# 写时拷贝的作用

写时拷贝的作用主要有三个。

## 第一，节省物理内存

多个进程拥有相同内容的页面时，可以先共享同一个物理页，而不是每个进程都保存一份。

## 第二，提高 fork 效率

fork 时不用立刻复制父进程的全部物理页。

操作系统只需要复制页表结构，并把相关页面标记成 COW。

## 第三，避免无意义复制

很多程序 fork 后马上 exec。

如果 fork 时立刻复制全部内存，那么这些复制很快就会被丢弃。

COW 可以避免这种浪费。

# 总结

写时拷贝本质上是：

> 操作系统通过页表权限位和 page fault 异常机制，把“立即复制整个地址空间”优化成“先共享物理页，等真正写入时再按页复制”。

它综合了两个思想：

```text
共享相同物理页，节省内存
+
写入时再复制，保证进程地址空间隔离
```

一句话理解：

> COW 的只读不是目的，而是手段。  
> 它用只读权限制造一个陷阱，让操作系统能在第一次写入时介入并完成复制。

再口语一点：

> 先假装复制了，其实底层还共享同一份物理页；等某个进程真的要写时，操作系统再偷偷复制一份给它。
