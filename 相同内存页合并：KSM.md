# 相同内存页合并：KSM

## 1. 为什么需要这个机制

有时候，不同进程之间会存在大量内容完全相同的内存页。

比如：

- 多个虚拟机运行相同操作系统；
    
- 多个进程加载了相同的数据；
    
- 同一个进程内部的不同虚拟页内容刚好相同。
    

如果这些内容相同的页在物理内存中各自保存一份，就会浪费很多内存。

所以操作系统可以把内容完全相同的页面合并起来，让多个虚拟页共同映射到同一个物理页，从而节省物理内存。

---

## 2. 基本过程

操作系统中的后台内核线程会扫描某些可合并的内存页。

如果发现两个或多个页面内容完全一样，就可以：

1. 保留其中一个物理页；
    
2. 修改相关进程的页表；
    
3. 让这些虚拟页都映射到同一个物理页；
    
4. 将这个共享物理页设置为只读；
    
5. 释放掉多余的物理页。
    

这样，从应用程序的角度来看，它原来访问自己的虚拟地址，现在还是访问同样的虚拟地址，程序并不知道底层物理页已经被合并了。

所以这个过程对用户态程序基本透明。

---

## 3. 为什么合并后的页要设置成只读

因为虽然这些页面现在内容相同，但之后不一定永远相同。

假设进程 A 和进程 B 的某个虚拟页都映射到了同一个物理页。

如果进程 A 想修改这个页面，而这个物理页仍然被大家共享，那么进程 A 的修改就会影响进程 B。

这显然不行。

所以操作系统会把合并后的共享页设置成只读。

一旦某个进程尝试写这个页面，CPU 会触发写保护异常，进入内核处理。

内核发现这是一个正常的写时拷贝场景，于是：

1. 分配一个新的物理页；
    
2. 把原共享页的内容复制到新物理页；
    
3. 修改当前进程的页表，让它映射到新物理页；
    
4. 给当前进程的新页面恢复写权限；
    
5. 让写操作继续执行。
    

这样，写入的进程获得了自己的副本，其他进程仍然共享原来的物理页。

这就是 **写时拷贝 COW**。

---

## 4. Linux 中的 KSM

Linux 中实现了这种机制，叫做：

```text
KSM：Kernel Samepage Merging
```

KSM 主要用于合并内容相同的匿名页，常见场景是虚拟化环境。

比如一台物理机上运行多个虚拟机，这些虚拟机可能都运行相同版本的 Linux，内存中就会有很多相同的页面。KSM 可以把这些相同页面合并，从而节省大量物理内存。

不过 KSM 不是无脑扫描所有物理内存。通常需要相关内存区域被标记为可以合并，内核线程才会扫描和尝试合并。

---

## 5. 缺点

KSM 的优点是节省内存，但它也有代价。

首先，扫描和比较页面本身需要 CPU 开销。

其次，如果合并后的共享页后来被频繁写入，就会不断触发：

```text
写保护异常 → 进入内核 → 分配新页 → 拷贝原页面 → 修改页表 → 返回用户态
```

这会带来明显的性能开销。

所以 KSM 适合：

```text
大量内容相同，并且之后主要读取、不频繁修改的页面
```

不适合：

```text
频繁写入、经常变化的页面
```

如果页面刚合并完马上又被写，那么合并节省的内存收益可能抵不过 COW 带来的开销。

---

## 6. 总结

相同内存页合并的核心思想是：

**如果多个虚拟页中的内容完全相同，就让它们共享同一个物理页，从而节省物理内存。**

为了防止一个进程写入共享页时影响其他进程，操作系统会把共享页设置为只读。

当某个进程真的要写这个页面时，就触发写保护异常，通过写时拷贝机制为它创建独立副本。

所以这个机制可以总结为：

```text
相同页面先共享；
需要写入时再复制。
```

Linux 中的 KSM 就是这种思想的具体实现。
