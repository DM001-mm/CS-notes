# 颠簸现象

## 1. 为什么会产生颠簸

颠簸现象指的是：系统大部分时间都花在处理缺页异常和等待磁盘 I/O 上，而真正用于执行程序指令的时间很少。

它的直接原因是：

**进程当前需要频繁访问的页面集合，也就是工作集，大于系统分配给它的物理页框数。**

换句话说，程序当前运行阶段需要这些页：

```text
A, B, C, D, E
```

但是系统只给了它 3 个页框。  
那么无论怎么换页，都很可能出现：

```text
刚换出去的页，马上又要访问
```

于是程序不断缺页，不断换入换出，CPU 很多时间都在等待磁盘，而不是执行真正有意义的计算。

---

## 2. 页替换算法为什么会和实际负载不匹配

页替换算法的目标是尽可能接近 MIN 算法。

MIN 算法的思想是：

> 淘汰未来最长时间不会被访问的页面。

但是现实中，操作系统不可能提前知道未来的访问序列，所以只能根据过去的信息进行猜测。

不同算法其实都有自己的假设：

|策略|它默认相信什么|
|---|---|
|FIFO|最早进入内存的页，可能最不需要了|
|LRU|最近访问过的页，接下来大概率还会继续访问|
|CLOCK|用访问位近似判断页面最近是否被访问过|
|LFU|过去访问次数多的页，以后可能还会继续被访问|

这些假设在有局部性的程序中通常有效。

比如程序在一段时间内集中访问某个数组、某段代码、某几个函数，那么最近访问过的页确实很可能还会继续访问。

但是如果程序的访问模式和算法假设不一致，就可能出现问题。

例如：

- 程序循环访问的页面数量大于物理页框数；
    
- 程序访问模式突然变化，旧的“最近访问页面”不再有用；
    
- FIFO 把重要页面换出去；
    
- LRU 被一次性扫描的大数组污染；
    
- 系统同时运行太多进程，每个进程分到的页框都不够。
    

这时页替换算法无法准确保留真正需要的页面，就会导致频繁缺页。

---

## 3. 调度器为什么可能让颠簸更严重

当进程发生缺页时，它需要等待磁盘把页面读入内存。  
在等待期间，这个进程不能继续运行，CPU 可能空闲下来。

操作系统调度器看到 CPU 利用率下降，可能会认为：

> CPU 没活干了，那我再调入更多进程运行吧。

但是问题在于，此时 CPU 利用率下降不是因为进程太少，而是因为内存压力太大，大家都在缺页等待磁盘。

如果系统继续增加进程数量，那么每个进程分到的物理页框可能更少，导致更多进程的工作集无法放入内存。

于是形成恶性循环：

```text
内存不够
→ 频繁缺页
→ 进程等待磁盘
→ CPU 利用率下降
→ 调度器增加并发进程
→ 每个进程分到的页框更少
→ 缺页更加严重
→ 系统进入颠簸
```

所以颠簸不是简单的“CPU太忙”，而是：

**CPU 看起来没什么有效工作可做，但系统整体却被缺页和磁盘 I/O 拖死了。**

# 工作集模型

## 1. 工作集的定义

一个进程在时刻 `t` 的工作集，指的是它在最近一段时间窗口 `[t - Δ, t]` 内访问过的页面集合。

记作：

```text
W(t, Δ)
```

其中：

- `t` 表示当前时刻；
    
- `Δ` 表示工作集窗口大小；
    
- `W(t, Δ)` 表示最近 Δ 时间内访问过的页面集合。
    

工作集模型的思想是：

> 程序在短时间内具有局部性。最近一段时间访问过的页，很可能在接下来一段时间还会继续被访问。

所以，工作集可以近似看作进程当前真正需要留在内存中的页面集合。

---

## 2. 工作集模型如何防止颠簸

如果一个进程的工作集大小是 5，那么系统最好至少给它 5 个物理页框。

如果给得太少，比如只给 3 个页框，那么这个进程就很容易频繁缺页。

所以操作系统可以根据工作集模型进行判断：

```text
如果所有进程的工作集总和 <= 可用物理页框数
    系统可以正常运行

如果所有进程的工作集总和 > 可用物理页框数
    系统可能发生颠簸
```

这时操作系统应该减少并发运行的进程数量，比如挂起一部分进程，把物理页框让给剩下的进程，让它们的工作集能够完整留在内存中。

这就是早期的 all-or-nothing 思想：

> 要么给一个进程足够的页框，让它的工作集能留在内存中；要么暂时不让它运行。

# 工作集时钟算法

## 1. 基本思想

工作集时钟算法，也叫 WSClock 算法。

它结合了两种思想：

```text
工作集思想 + CLOCK 算法
```

普通 CLOCK 算法主要看页面的访问位：

- 访问位为 1：说明最近访问过，给第二次机会；
    
- 访问位为 0：说明最近没访问，可以考虑淘汰。
    

但是普通 CLOCK 算法只知道“最近有没有访问”，不知道“多久没访问”。

工作集时钟算法在此基础上给每个页面增加一个信息：

```text
上次使用时间
```

这样它就可以判断：

> 这个页面是不是已经很久没有被访问了？  
> 如果很久没访问，说明它可能已经不在当前工作集中了，可以淘汰。

---

## 2. 每个页面需要维护的信息

工作集时钟算法通常需要维护：

|信息|作用|
|---|---|
|访问位 R|判断页面最近是否被访问过|
|修改位 M|判断页面是否被写过|
|上次访问时间|判断页面是否还属于工作集|
|当前时刻|用来计算页面年龄|
|工作集窗口 τ|判断页面是否太久没被访问|

页面年龄可以理解为：

```text
页面年龄 = 当前时间 - 上次访问时间
```

如果页面年龄大于 τ，说明它已经很久没被访问，可能不属于当前工作集。

---

## 3. 算法过程

工作集时钟算法仍然像 CLOCK 一样，用一个指针在页面表中循环扫描。

扫描到某个页面时，分几种情况讨论。

### 情况一：访问位 R = 1

说明这个页面最近被访问过。

那么：

```text
把 R 置为 0
更新该页面的上次访问时间
指针继续向前走
```

这个页面暂时不淘汰，因为它还可能属于当前工作集。

---

### 情况二：访问位 R = 0，而且页面年龄 <= τ

说明这个页面虽然这一轮没有被访问，但它上次访问距离现在还不算久。

也就是说，它可能仍然在工作集里面。

所以：

```text
不淘汰
指针继续向前走
```

---

### 情况三：访问位 R = 0，页面年龄 > τ，而且修改位 M = 0

说明：

- 最近没访问；
    
- 很久没访问；
    
- 没有被修改过，不需要写回磁盘。
    

这种页面就是理想淘汰对象。

所以：

```text
直接淘汰该页面
```

---

### 情况四：访问位 R = 0，页面年龄 > τ，但是修改位 M = 1

说明：

- 最近没访问；
    
- 很久没访问；
    
- 但是被修改过。
    

这种页面虽然适合淘汰，但不能直接丢掉，因为它是脏页，必须先写回磁盘。

所以：

```text
安排该页面写回磁盘
指针继续向前找别的页面
```

等它写回完成后，之后再扫描到它，就可以淘汰了。

---

## 4. 工作集时钟算法相比 CLOCK 多了什么

普通 CLOCK 算法只看访问位：

```text
最近有没有访问过？
```

工作集时钟算法还要看时间：

```text
这个页面多久没被访问了？
它是否已经不属于当前工作集？
```

所以它比 CLOCK 更接近工作集模型。

可以这样理解：

```text
CLOCK：给最近访问过的页面第二次机会。

WSClock：不仅给最近访问过的页面机会，还会判断页面是否已经老到离开了工作集。
```

---

## 5. 总结

颠簸现象的本质是：

**进程需要的工作集装不进分配给它的物理页框，导致频繁缺页。**

页替换算法的作用是尽可能保留未来还会访问的页面，但它只能根据过去的信息推测未来。

工作集模型认为：

**最近一段时间访问过的页面，很可能就是当前阶段真正需要的页面。**

工作集时钟算法就是在 CLOCK 算法的基础上加入“上次访问时间”，用访问位和时间共同判断页面是否还属于工作集，从而尽量减少错误淘汰，缓解颠簸现象。
