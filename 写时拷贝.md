# 为什么

现在考虑两个场景:
第一个场景是两个应用程序拥有很多相同的内存数据（比如加载了相同的动态链接库)如果把这些数据相同的内存页在物理内存中仅存一份并以只读的方式映射给两个应用程序那么就能够显著的节约物理内存资源。
第二个场景是在Linux操作系统中一个应用程序可以用fork系统调用创建子程序。初始时，父子程序的全部内存数据和地址空间完全一样如何能够高效地实现这种应用程序创建机制呢？

等讲了这个写时拷贝作用过程 ，你就知道他综合了上面两种场景。而且，优点也是第一个场景的优点

# CPU 硬件实现

页表项 中不止有物理页的地址，还有一些属性位，在学的时候 ，你不是对 读/写 和 是否可执行 有些分不清吗 解释->[[页表项中的读写和可执行 属性位]]

就是用了 读/写 这个属性位

# 逻辑过程
多个进程拥有 的内存页中的 数据 相同，就考虑直接用一个 物理页 来存储，多个进程的页表都映射到 这一个物理地址，然后 将相应的页表项中的读写位设置成只读，如果有进程需要写，（就会用到第二个场景中的fork~~到这里你应该就知道了~~）然后 MMU 查询页表 ，发现这个页表项是 只读，就会报缺页异常，然后 陷入(系统调用，~~就是异常处理~~），操作系统相应的异常处理函数识别出这个 物理页 是写时拷贝，于是，操作系统会重新拷贝一份，完成映射，add_mapping ,添加TLB，但是这次的 读写位 设置成 可读可写，然后回到异常发生处，重新运行该指令

# 作用

这个会出现再后面的物理内存不够，通过虚拟内存优化物理内存 那的方法中

# 为什么有理解上的矛盾

> 针对刚刚提出的两个场景，写时拷贝技术一方面能够节约物理内存资源，比如，不同的应用程序以写时拷贝的方式映射相同动态链接库的可写部分（动态链接库的只读部分以只读方式在不同应用程序间共享即可）；另一方面，运用写时拷贝可以让父子程序以只读的方式共享全部内存数据避免内存拷贝操作带来的时间和空间的开销

反正写时拷贝 设置只读 是假的只读，会在操作系统相应的异常处理函数中被识别，到那时 才会映射物理地址，设置 可读可写，不看中间 等价于 可写，所以，没有矛盾
